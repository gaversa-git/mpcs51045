# Controller Area Network (CAN) Message Decoder and Encoder Framework

This project is a small modern C++ framework for encoding and decoding automotive-style CAN messages. The idea is that raw CAN frames are represented by the 'frame' type in 'include/can/frame.h', while specific vehicle messages are represented as strongly typed C++ structs in 'messages/vehicle_messages.h'. For example, the project includes messages for vehicle speed, steering angle, brake pressure, battery temperature, and individual wheel speeds. Each message knows how to encode itself into an 8-byte CAN payload and how to decode itself back from that payload. The demo in 'src/main.cpp' shows this full pipeline: creating typed messages, converting them into raw frames, printing them in a simple candump-style format, parsing frames back from that format, decoding them through a registry, and dispatching the decoded result to handlers.

The main goal of the project was not to build a fully production-ready CAN or DBC parser, but to use a realistic embedded/automotive-flavored problem as a way to apply techniques from the course. The project uses concepts in 'message_concept.h' to define what counts as a valid CAN message, so generic code can reject invalid message types at compile time. It uses templates and 'constexpr' logic in 'signal.h' to define packed bit fields and catch invalid signal layouts early. It uses variadic templates and fold expressions in 'registry.h' to build a compile-time registry of supported message types, which avoids writing a large manual switch statement over CAN IDs. It also uses 'std::variant' and visitors to represent decoded messages without losing their concrete type. Other course topics appear in smaller places too, such as custom hashing for 'can::frame' in 'hash.h', typelist-style compile-time utilities in 'typelist.h', CRTP-style static dispatch in 'static_dispatch.h', and a PMR-backed telemetry buffer in 'pmr_buffer.h'.

Architecturally, I tried to keep the core types simple and value-oriented. A CAN frame is just a small value object, and the vehicle messages are also plain structs rather than being part of an inheritance hierarchy. I chose this because it feels closer to embedded C++ style: the data types are small, explicit, and easy to copy or inspect. The more advanced C++ machinery is kept in the framework layer instead of being scattered through the message definitions. For example, 'codec.h' handles the generic encode/decode functions, while 'registry.h' handles the compile-time mapping from frame IDs to concrete message types. This separation makes the message definitions readable while still allowing the framework to demonstrate concepts, variants, folds, and generic programming.

I built the project with CMake as a mostly header-only library. From the project root, it can be configured and built with 'cmake -S . -B build' followed by 'cmake --build build'. The main executables are 'can_demo', 'test_phase1', and 'test_roundtrip'. 'test_phase1' checks the basic typed encode/decode path, while 'test_roundtrip' checks the fuller version of the project, including the registry, unknown ID handling, log parsing, hashing, typelist checks, CRTP dispatch, and the PMR buffer. The intended result is that both tests pass and the demo prints the raw frames, decoded messages, parsed log frames, handler counts, and buffer/hash demonstrations.

I considered a few alternative designs and rejected them for this version. One option was to make every message inherit from a common virtual base class, but that would have pushed the design toward runtime polymorphism and made the message types heavier than necessary. Since CAN messages are small fixed-format value types, compile-time polymorphism with concepts felt more appropriate. Another option was to use a runtime 'unordered_map' from CAN ID to decoder functions. That would be more flexible if message definitions were loaded dynamically, but it would not demonstrate as much of the compile-time template material from the course, and it would make the decoded type harder to preserve cleanly. I also considered writing a more complete DBC parser, but that would have turned the project into more of a parsing assignment than an advanced C++ design project. Instead, I kept the log format deliberately small and focused the effort on the type-safe encoding, registry, dispatch, and memory/ownership choices.
